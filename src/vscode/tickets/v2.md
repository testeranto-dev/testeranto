## functionality

We want testeranto to have it's own space int he activity bar. Right now, it's living under the files section, but we want out own dedicated space.

Our extensions will look similar to the basic files section. the left column will show a tree, and the right side will show file contents.

This element differs from the vanilla file explorerer in several ways:

- It does not show ALL the files in the repo, it shows only the
  -- globbed documentation files
  -- input files for tests
  -- reports and a few other key files TBD

- each test has a button that when clicked opens a terminal tab in vscode

- produce an html page with the src folder, with only the features and test results, but no source code. This is for non-coder stakeholders. It's most important goal is to render a tree congruent with our source code structure, and with that tree, show test results and features, but no iimplementation details. We'll need to be able to check this file into git as the testeranto/reports/index.html, because it will be a static website on github. We need to show this somewhere in the vscode extension.

1. A monorepo project has many runtimeConfig
2. runtimeConfigs are pieces of your code base pertaining to a particular langauge.

example:
the runtimeConfigKey is "nodetests" but the runtime is "node"

```
    nodetests: (
      {
        runtime: "node",
        tests: ["src/ts/Calculator.test.node.ts"],
        checks: [
          (x) => `yarn eslint ${x.join(' ')} `,
          (x) => `yarn tsc --noEmit ${x.join(' ')}`,
        ],
        dockerfile: `testeranto/runtimes/node/node.Dockerfile`,
        buildOptions: `testeranto/runtimes/node/node.mjs`,
        buildKitOptions: {
          // Single-stage Dockerfile, no targetStage needed
        },
        outputs: [],

      }
    ),
```

3. Each runtime config has tests.
4. Each tests has features.
5. features can be string, other files, or urls
6. Every bdd test and every check is a docker container process.
7. Every bdd test and every check produces logs
8. Every bdd test produces a test.json file, which is a summarization of pass/fails

### How can we present this?

1. As a tree. Recreate the file treee with just the reports, docs, and input files.
2. As a list of runtimeConfigs, then broken down by test, then by feature
3. A list of processes. Each bdd and check test is a docker process.
